This presentation introduced my initial study on Automated Program Repair and patch correctness assessment.

Automated Program Repair aims to automatically generate patches for software bugs. However, a generated patch that passes the existing test suite is not always semantically correct. Some patches only satisfy the available tests while still violating the intended behavior of the program. This problem is often referred to as the overfitting patch problem.

The presentation focused on why patch correctness assessment is important after a plausible patch is generated. I reviewed the basic motivation of learning-based patch correctness prediction and discussed how existing approaches use code changes, program representations, and model-based classification to distinguish correct patches from overfitting patches.

This talk also served as a starting point for my current research direction: exploring whether execution-aware signals can provide additional evidence for patch correctness assessment. Instead of only comparing source code before and after a patch, this direction considers how the patched program behaves during execution.
